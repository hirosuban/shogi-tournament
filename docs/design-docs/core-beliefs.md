# Core Beliefs

**ステータス**: ✅ Active
**日付**: 2026-03-21

このドキュメントはエージェントファーストの開発における中核的信念を定義します。
すべての設計判断・実装判断はこれらの信念と整合している必要があります。

---

## 1. リポジトリが唯一の信頼できる情報源

エージェントの実行時コンテキスト内でアクセスできないものは、存在しないも同然です。

- Google Docs・Slack・個人の頭の中にある知識はエージェントには見えない
- すべての重要な知識はリポジトリ内にバージョン管理されたアーティファクトとして存在する
- 意思決定・アーキテクチャの合意・チームの規約はすべて `docs/` にエンコードする

## 2. エージェントにはマップを与える。百科事典ではなく

- AGENTS.md は目次として機能する（~100行程度）
- 詳細は各専門ドキュメントに委ねる
- コンテキストは希少なリソース。重要でない情報でコンテキストを埋めない

## 3. 段階的開示

- エージェントは小さく安定した入口から始める
- 最初から膨大な情報に圧倒させない
- 必要に応じてより深い情報源へのナビゲーションを提供する

## 4. 機械的に適用されるルールを優先する

- ドキュメントだけでは一貫性を保てない
- 重要な不変条件はリンター・構造テスト・CIで機械的に適用する
- リンターのエラーメッセージには修復手順を含める

## 5. 「退屈」な技術を優先する

- 予測可能で安定したAPIを持つ技術を選ぶ
- エージェントが推論しやすい（学習データに豊富に含まれる）技術を優先する
- 不透明な上流依存を避ける。必要なら機能を再実装する

## 6. エントロピーをガベージコレクションする

- コードベースのドリフトは定期的にクリーンアップする
- 技術的負債は高金利のローン。早めに少しずつ返済する
- 「品質整備」エージェントを定期実行して陳腐化を検出する

## 7. 人間の役割はシステムの設計

- 人間はコードを書くのではなく、環境・抽象化・フィードバックループを設計する
- エージェントが行き詰まったらツールやガードレールが不足しているシグナル
- 不足しているものをエージェントに修正させる

## 8. スループットを最大化するためにブロックを最小化する

- PRは小さく・短命に保つ
- テストのフレークはフォローアップで対処（PRをブロックしない）
- 待機コストはシステム全体では修正コストより高い
