# Phase 4: デプロイ・仕上げ

## 目標

フロントエンドとバックエンドを本番環境にデプロイし、継続的に動作する状態にする。

## 背景・動機

Phase 1〜3が完了し、ローカルで動作する状態になったら着手する。

## アプローチ

- フロントエンド: Vercel（静的ファイルのため無料枠で十分）
- バックエンド: Render または Fly.io（**暫定。要件・コストによって変更可能性あり**）
- スクレイパーの定期実行はバックエンドと同一プロセスで動作

## タスク

- [ ] フロントエンドのVercelデプロイ設定（`vercel.json` or Vercel Dashboard）
- [ ] バックエンドのデプロイ設定（Dockerfileまたはビルド設定）
- [ ] 環境変数の整理（開発/本番の切り替え）
- [ ] フロントエンドのAPI URLを本番エンドポイントに切り替え
- [ ] CORSの本番オリジン設定
- [ ] デプロイ後の動作確認（スクレイピング・API・画面表示が正常に動くこと）
- [ ] スクレイパーの定期実行が本番で動作していることを確認

## 意思決定ログ

- ホスティング先は暫定。変更した場合は `ARCHITECTURE.md` と `docs/design-docs/tech-stack.md` を同じPRで更新すること

## 完了条件

- 本番URLでアプリが閲覧できること
- スクレイパーが定期的に動作し、データが更新されていること
